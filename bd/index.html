Conceitos Básicos de Banco de Dados

O que é um Banco de Dados?

Um banco de dados é uma coleção organizada de dados que podem ser facilmente acessados, gerenciados e atualizados. Pense nele como uma biblioteca digital onde as informações são armazenadas de forma estruturada.

Exemplo: Uma lista de contatos no seu celular é um banco de dados simples.

Características Principais

  • Organização: Dados estruturados de forma lógica
  • Consistência: Informações confiáveis e sem contradições
  • Compartilhamento: Acesso simultâneo por múltiplos usuários
  • Persistência: Dados armazenados permanentemente

Curiosidade

O primeiro sistema de banco de dados comercial foi desenvolvido pela IBM na década de 1960, chamado IMS (Information Management System). Desde então, a evolução dos bancos de dados revolucionou a forma como armazenamos e acessamos informações.

Banco de Dados, SGBD e Sistema de Banco de Dados

Banco de Dados (BD)

É o conjunto de dados em si, armazenados de forma organizada. Não inclui o software para gerenciá-lo.

Analogia: Os livros em uma biblioteca

SGBD

Sistema de Gerenciamento de Banco de Dados - Software que gerencia o BD, permitindo criar, consultar, atualizar e administrar os dados.

Analogia: O bibliotecário e o sistema de catalogação

Sistema de BD

Conjunto completo incluindo o BD, o SGBD, os usuários, aplicativos e procedimentos.

Analogia: A biblioteca inteira com funcionários e regras

Vantagens de Usar um SGBD

Segurança de Dados

Controle de acesso granular com diferentes níveis de permissão para usuários, garantindo que apenas pessoas autorizadas possam acessar ou modificar informações sensíveis.

Exemplo: Um funcionário do RH pode ver salários, mas um vendedor não.

Concorrência e Multiusuário

Permite que múltiplos usuários acessem e modifiquem os dados simultaneamente sem causar inconsistências, através de mecanismos de controle de concorrência.

Exemplo: Vários caixas de um supermercado podem atualizar o estoque ao mesmo tempo.

Integridade de Dados

Garante que os dados sigam regras de negócio e restrições definidas, prevenindo entrada de informações inválidas ou inconsistentes.

Exemplo: Impedir que um produto tenha preço negativo ou estoque fracionário.

Independência de Dados

Permite modificar a estrutura do banco de dados sem afetar as aplicações que o utilizam, facilitando manutenção e evolução do sistema.

Exemplo: Adicionar um novo campo na tabela de clientes sem quebrar o sistema.

Performance Otimizada

Algoritmos de otimização de consultas, indexação automática e cache de dados garantem respostas rápidas mesmo com grandes volumes de informação.

Exemplo: Buscar um cliente entre milhões em milissegundos.

Recuperação de Falhas

Mecanismos de backup e recuperação automáticos garantem que os dados possam ser restaurados em caso de falhas do sistema ou desastres.

Exemplo: Restaurar o banco para um estado consistente após uma queda de energia.

Campos, Registros e Tabelas

Estes são os elementos fundamentais que compõem a estrutura de um banco de dados relacional. Vamos entender cada um deles com exemplos práticos.

Campos (Colunas)

São as unidades básicas de informação que representam características específicas dos dados.

  • • Tipo de dado definido
  • • Nome único na tabela
  • • Podem ter restrições

Registros (Linhas)

Cada registro representa uma instância completa de dados, contendo valores para todos os campos.

  • • Conjunto de valores
  • • Representa um item
  • • Deve ser único

Tabelas

Estrutura que organiza dados em linhas e colunas, representando uma entidade do mundo real.

  • • Nome único no BD
  • • Define a estrutura
  • • Armazena registros
ID Nome Email Idade Cidade
1 João Silva joao@email.com 28 São Paulo
2 Maria Santos maria@email.com 32 Rio de Janeiro
3 Pedro Oliveira pedro@email.com 25 Belo Horizonte

Exemplo SQL - Criar Tabela 

CREATE TABLE clientes (
    id INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE,
    idade INT CHECK (idade >= 0),
    cidade VARCHAR(50)
);

Exemplo SQL - Inserir Dados 

INSERT INTO clientes (nome, email, idade, cidade)
VALUES ('João Silva', 'joao@email.com', 28, 'São Paulo'),
       ('Maria Santos', 'maria@email.com', 32, 'Rio de Janeiro'),
       ('Pedro Oliveira', 'pedro@email.com', 25, 'Belo Horizonte');

Índices

Índices são estruturas de dados que melhoram significativamente a velocidade de operações de recuperação de dados em uma tabela. Funcionam como o índice de um livro, permitindo encontrar informações rapidamente sem precisar ler tudo.

Benefícios dos Índices

  • Busca Rápida: Reduz drasticamente o tempo de consultas
  • Ordenação Eficiente: Acelera operações ORDER BY
  • Junções Otimizadas: Melhora performance de JOINs
  • Unicidade Garantida: Índices únicos evitam duplicatas

Desvantagens

  • Espaço Extra: Consome armazenamento adicional
  • Inserções Lentas: INSERT, UPDATE e DELETE ficam mais lentos
  • Manutenção: Precisa ser atualizado com cada modificação
  • Complexidade: Índices mal planejados podem piorar performance

Como Funciona um Índice?

Sem Índice

Busca linear: O banco lê cada registro

Registro 1: ID=100
Registro 2: ID=250
Registro 3: ID=375 ← Encontrado!
Registro 4: ID=500

Tempo: O(n) - Linear

Com Índice

Busca binária: Pula diretamente ao registro

Índice: 100 → Registro 1
Índice: 250 → Registro 2
Índice: 375 → Registro 3 ← Encontrado!
Índice: 500 → Registro 4

Tempo: O(log n) - Logarítmico

Criar Índice 

-- Índice simples
CREATE INDEX idx_cliente_nome 
ON clientes(nome);

-- Índice composto
CREATE INDEX idx_cliente_cidade_idade 
ON clientes(cidade, idade);

-- Índice único
CREATE UNIQUE INDEX idx_cliente_email 
ON clientes(email);

Quando Usar Índices?

  • • Colunas frequentemente usadas em WHERE
  • • Colunas de JOIN entre tabelas
  • • Colunas em ORDER BY frequentes
  • • Colunas com alta cardinalidade (muitos valores únicos)
  • • Evitar em tabelas com muitas atualizações

Tipos de Chaves

Chaves são colunas ou combinações de colunas usadas para identificar exclusivamente registros em uma tabela e estabelecer relacionamentos entre tabelas.

Chave Primária (PK)

Identifica exclusivamente cada registro na tabela. Deve ser única e não nula.

Características:

  • • Única
  • • Não nula
  • • Imutável (geralmente)

Chave Estrangeira (FK)

Referencia a chave primária de outra tabela, estabelecendo relacionamentos.

Características:

  • • Referencial
  • • Mantém integridade
  • • Pode ser nula

Chave Candidata

Qualquer coluna ou conjunto que poderia servir como chave primária.

Características:

  • • Única
  • • Não nula
  • • Múltiplas por tabela

Chave Única

Garante que os valores em uma coluna sejam únicos, mas não é a chave primária.

Características:

  • • Única
  • • Pode ser nula
  • • Múltiplas por tabela

Chave Composta

Chave primária formada por duas ou mais colunas.

Características:

  • • Múltiplas colunas
  • • Combinação única
  • • Usada em tabelas de junção

Chave Surrogate

Chave primária artificial, sem significado de negócio (ex: auto-incremento).

Características:

  • • Artificial
  • • Numérica
  • • Sem significado

Exemplo Prático de Relacionamento

Tabela: Clientes
id (PK) nome email
1 João Silva joao@email.com
2 Maria Santos maria@email.com
Tabela: Pedidos
id (PK) cliente_id (FK) valor
101 1 R$ 150,00
102 2 R$ 200,00
103 1 R$ 75,00

Relacionamento: Um cliente pode ter vários pedidos (1:N)

SQL com Chaves 

-- Tabela com chave primária
CREATE TABLE clientes (
    id INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE
);

-- Tabela com chave estrangeira
CREATE TABLE pedidos (
    id INT PRIMARY KEY AUTO_INCREMENT,
    cliente_id INT,
    valor DECIMAL(10,2),
    FOREIGN KEY (cliente_id) 
    REFERENCES clientes(id)
);

Boas Práticas

  • • Use chaves surrogate (auto-incremento) como PK
  • • Mantenha chaves estrangeiras consistentes
  • • Crie índices em chaves estrangeiras
  • • Use nomes significativos para as chaves
  • • Evite chaves compostas quando possível

Tipos de Integridade

Integridade de dados refere-se à precisão, consistência e confiabilidade dos dados armazenados no banco de dados. Existem vários tipos de integridade que garantem a qualidade das informações.

Integridade de Entidade

Garante que cada tabela tenha uma chave primária e que todos os valores na chave primária sejam únicos e não nulos. Nenhum registro pode ser duplicado.

Exemplo:

CREATE TABLE funcionarios (
    id INT PRIMARY KEY,  -- Garante integridade de entidade
    nome VARCHAR(100) NOT NULL,
    salario DECIMAL(10,2)
);

-- Tentativa de inserir ID duplicado falhará
INSERT INTO funcionarios VALUES (1, 'João', 5000);
INSERT INTO funcionarios VALUES (1, 'Maria', 6000); -- ERRO!

Integridade Referencial

Assegura que os relacionamentos entre tabelas permaneçam consistentes. Uma chave estrangeira deve sempre referenciar uma chave primária válida existente.

Exemplo:

CREATE TABLE departamentos (
    id INT PRIMARY KEY,
    nome VARCHAR(50)
);

CREATE TABLE funcionarios (
    id INT PRIMARY KEY,
    nome VARCHAR(100),
    depto_id INT,
    FOREIGN KEY (depto_id) 
    REFERENCES departamentos(id)
);

-- Tentativa de inserir funcionário com depto inexistente falha
INSERT INTO funcionarios VALUES (1, 'João', 99); -- ERRO!

Integridade de Domínio

Especifica que todos os valores em uma coluna devem pertencer a um domínio definido, incluindo tipo de dado, formato, intervalo de valores e outras restrições.

Exemplo:

CREATE TABLE produtos (
    id INT PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    preco DECIMAL(10,2) CHECK (preco > 0),
    estoque INT DEFAULT 0 CHECK (estoque >= 0),
    categoria VARCHAR(20) 
        CHECK (categoria IN ('Eletrônico', 'Vestuário', 'Alimentos'))
);

-- Valores inválidos serão rejeitados
INSERT INTO produtos VALUES 
    (1, 'Celular', -100, 10, 'Eletrônico'); -- ERRO: preço negativo
INSERT INTO produtos VALUES 
    (2, 'Camisa', 50, -5, 'Vestuário');    -- ERRO: estoque negativo

Integridade de Definida pelo Usuário

Regras de negócio complexas que não podem ser expressas através das integridades anteriores. Geralmente implementadas através de triggers, stored procedures ou validações na aplicação.

Exemplo com Trigger:

CREATE TRIGGER verificar_salario_minimo
BEFORE INSERT ON funcionarios
FOR EACH ROW
BEGIN
    DECLARE salario_min DECIMAL(10,2);
    SELECT valor INTO salario_min 
    FROM parametros_sistema 
    WHERE nome = 'salario_minimo';
    
    IF NEW.salario < salario_min THEN
        SIGNAL SQLSTATE '45000' 
        SET MESSAGE_TEXT = 'Salário abaixo do mínimo permitido';
    END IF;
END;

Ações de Integridade Referencial

CASCADE

Ao deletar/atualizar o registro pai, todos os registros filhos correspondentes são também deletados/atualizados.

SET NULL

Ao deletar/atualizar o registro pai, os registros filhos têm sua chave estrangeira definida como NULL.

RESTRICT

Impede a deleção/atualização do registro pai se existirem registros filhos referenciando-o.

Exercícios Práticos

Exercício 1: Criando um Banco de Dados de Biblioteca

Crie as tabelas necessárias para um sistema de biblioteca que armazene informações sobre livros, autores e empréstimos.

Requisitos:

  • Cada livro tem um título, ISBN, ano de publicação e um autor
  • Cada autor tem nome, nacionalidade e data de nascimento
  • Registre empréstimos com data de empréstimo e devolução prevista
  • Um livro pode ter apenas um empréstimo ativo por vez
-- Tabela de Autores
CREATE TABLE autores (
    id INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(100) NOT NULL,
    nacionalidade VARCHAR(50),
    data_nascimento DATE
);

-- Tabela de Livros
CREATE TABLE livros (
    id INT PRIMARY KEY AUTO_INCREMENT,
    titulo VARCHAR(200) NOT NULL,
    isbn VARCHAR(20) UNIQUE NOT NULL,
    ano_publicacao INT,
    autor_id INT,
    FOREIGN KEY (autor_id) REFERENCES autores(id)
);

-- Tabela de Empréstimos
CREATE TABLE emprestimos (
    id INT PRIMARY KEY AUTO_INCREMENT,
    livro_id INT,
    data_emprestimo DATE NOT NULL,
    data_devolucao_prevista DATE NOT NULL,
    data_devolucao_real DATE,
    FOREIGN KEY (livro_id) REFERENCES livros(id),
    CHECK (data_devolucao_prevista > data_emprestimo)
);

-- Índice para melhorar performance de buscas
CREATE INDEX idx_livros_titulo ON livros(titulo);
CREATE INDEX idx_emprestimos_livro ON emprestimos(livro_id);

Exercício 2: Consultas SQL

Usando as tabelas criadas no exercício anterior, escreva consultas SQL para responder às seguintes perguntas:

a) Liste todos os livros publicados após 2010:

SELECT titulo, ano_publicacao 
FROM livros 
WHERE ano_publicacao > 2010
ORDER BY ano_publicacao DESC;

b) Encontre todos os livros de autores brasileiros:

SELECT l.titulo, a.nome AS autor
FROM livros l
JOIN autores a ON l.autor_id = a.id
WHERE a.nacionalidade = 'Brasileira';

c) Liste livros com empréstimos em atraso:

SELECT l.titulo, e.data_devolucao_prevista,
       DATEDIFF(CURDATE(), e.data_devolucao_prevista) AS dias_atraso
FROM emprestimos e
JOIN livros l ON e.livro_id = l.id
WHERE e.data_devolucao_real IS NULL 
  AND e.data_devolucao_prevista < CURDATE();

Exercício 3: Desafio de Normalização

A tabela abaixo contém informações de vendas. Identifique os problemas de design e normalize para 3FN (Terceira Forma Normal).

ID_Venda Data Cliente Endereço Cliente Produto Categoria Quantidade Preço Unitário
1 2024-01-15 João Silva Rua A, 123 Notebook Eletrônico 1 3500.00

Problemas identificados:

  • Redundância de dados (cliente e endereço repetidos)
  • Anomalias de atualização (mudar endereço do cliente em múltiplos lugares)
  • Dependências parciais (categoria depende apenas do produto)

Tabelas normalizadas (3FN):

-- Tabela Clientes
CREATE TABLE clientes (
    id INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(100) NOT NULL,
    endereco VARCHAR(200)
);

-- Tabela Categorias
CREATE TABLE categorias (
    id INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(50) UNIQUE NOT NULL
);

-- Tabela Produtos
CREATE TABLE produtos (
    id INT PRIMARY KEY AUTO_INCREMENT,
    nome VARCHAR(100) NOT NULL,
    categoria_id INT,
    preco_unitario DECIMAL(10,2),
    FOREIGN KEY (categoria_id) REFERENCES categorias(id)
);

-- Tabela Vendas
CREATE TABLE vendas (
    id INT PRIMARY KEY AUTO_INCREMENT,
    data DATE NOT NULL,
    cliente_id INT,
    FOREIGN KEY (cliente_id) REFERENCES clientes(id)
);

-- Tabela Itens_Venda
CREATE TABLE itens_venda (
    venda_id INT,
    produto_id INT,
    quantidade INT NOT NULL,
    PRIMARY KEY (venda_id, produto_id),
    FOREIGN KEY (venda_id) REFERENCES vendas(id),
    FOREIGN KEY (produto_id) REFERENCES produtos(id)
);
